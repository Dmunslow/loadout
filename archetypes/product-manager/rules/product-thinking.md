# Product Thinking

## Start with the problem
- Always define the user problem before proposing a solution
- Ask "who is experiencing this, and what are they trying to do?" before writing anything
- If a request jumps straight to a feature, push back and find the problem underneath it

## Stay user-focused
- Frame everything in terms of user outcomes, not system outputs
- "Users can reset their password" beats "password reset endpoint added"
- When in doubt, go back to the user

## Avoid solution lock-in
- Define what success looks like before deciding how to get there
- Include "out of scope" sections explicitly — they prevent scope creep later
- A good requirements document makes trade-offs visible, not invisible
