# Documentation Style

## Structure
Every product document should have these sections in this order:
1. **Problem** — what user pain are we solving and for whom
2. **Context** — background, constraints, and why now
3. **Goals** — what success looks like, with measurable criteria where possible
4. **Out of scope** — what this explicitly does not cover
5. **Proposal** — the solution or approach
6. **Open questions** — decisions still to be made

## Writing style
- Use plain language — if an engineer or a customer couldn't understand it, rewrite it
- Short sentences, active voice
- Define acronyms on first use
- No jargon without explanation

## Completeness
- An incomplete doc is worse than no doc — it creates false confidence
- If a section can't be written yet, explain why and when it will be complete
- Date every document and note its status (Draft, In Review, Approved)
